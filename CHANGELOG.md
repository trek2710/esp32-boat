- **2026-04-23** — Display bring-up round 10: CHIPSET PIVOT. Round 9's exhaustive 420-pair I2C scan produced a clean result — only SDA=GPIO15 ever ACKs, and the only three addresses that respond are 0x15 (CST820 touch), 0x20 (TCA9554 I/O expander), and 0x51 (PCF85063 RTC). 0x24 and 0x38 — the CH422G MODE + WR_IO registers — never ACKed on any pin pair, meaning the board does NOT host a CH422G at all. That address signature (0x15 / 0x20 / 0x51) matches Waveshare's *other* 'ESP32-S3-Touch-LCD-2.1' BOM: ST7701 RGB panel (NOT QSPI ST77916) + TCA9554 expander (NOT CH422G) + CST820 + PCF85063. Every one of our last 9 rounds has been building drivers for the wrong chipset family. This round does the minimum useful thing with that information: (1) lock I2C pins in display_pins.h to SDA=GPIO15 / SCL=GPIO7 (GPIO7 is Waveshare's silkscreen convention on the 4' sibling board; it gets verified at boot), (2) document the chipset pivot inline with a pivot plan (round 11 = swap ch422g.cpp for TCA9554; round 12+ = replace ST77916 QSPI driver with esp_lcd_panel_rgb ST7701 + PSRAM framebuffer), (3) add TCA9554 register/bit definitions (TCA9554_REG_CONFIG=0x03, TCA9554_REG_OUTPUT=0x01, bits IO0=LCD_BL / IO1=TP_RST / IO2=LCD_RST / IO3=SD_CS) alongside the existing CH422G ones — the CH422G symbols stay temporarily so ch422g.cpp still links, but are clearly marked UNUSED, (4) swap round 9's 420-pair exhaustive scan for a targeted round-10 diagnostic that fixes SDA=GPIO15 and runs the full 127-address scan on each SCL candidate in {7, 1, 2, 5, 8, 9, 16}. The correct SCL will produce exactly three ACKs (0x15 / 0x20 / 0x51); wrong SCLs will drop ACKs or produce spurious ones. Once round 10's serial log confirms SCL=GPIO7 gives the clean 3-hit set, round 11 can rip out ch422g.cpp and move on to the ST7701 RGB rewrite. No driver code changes this round — purely diagnostic + pivot documentation.
- **2026-04-22** — Display bring-up round 6: I2C is still blind on GPIO8/9 (both orderings) AND GPIO10/11 (both orderings) — we've exhausted the plausible guesses derived from other Waveshare CH422G boards. Stop guessing, start diagnosing: add a probeIdle() in Ui.cpp::begin() that, BEFORE Wire.begin(), drives pins as INPUT_PULLUP, waits 200 µs, and reads them. Both HIGH = bus electrically healthy (wrong pins or dead device). Either LOW = bus shorted / missing 3V3 / stuck peripheral — in which case no amount of pin guessing will help. We probe three candidate pairs in one flash (8/9, 10/11, 6/7) so one serial dump tells us which GPIOs even look like a live I2C bus on this particular board. SPI/QSPI side still unchanged — SPI3_HOST, 64-byte sendPixels chunking, minimal init sequence. Expectation for this flash: the idle-state log narrows 'pin problem' vs 'electrical problem' so we know whether to keep trying pins or pull out a multimeter.
- **2026-04-22** — Display bring-up round 4: SPI side now clean — 64-byte sendPixels chunking, SPI3_HOST, and dropping spi_bus_free made runInitSequence/backlight-on/fillColor go through with no 'txdata transfer > hardware max supported len' errors on the panel. I2C side still blind: scan sees zero devices at 100 kHz with pins (SDA=11, SCL=10). That leaves pin-swap as the strongest remaining candidate — Waveshare has shipped multiple ESP32-S3-Touch-LCD-2.1 revisions with SDA/SCL swapped. So Ui.cpp::begin() now tries the documented pin order first, scans; if zero devices, calls Wire.end() and re-inits with pins swapped, scans again. Whichever ordering ACKs devices is the one we keep. Also moved the scan out of Ch422g::begin() (now run once in Ui.cpp before the expander is touched) and dropped I2C to 100 kHz to tolerate the ESP32's weak internal pull-ups in case there are no board-level pull-ups.
