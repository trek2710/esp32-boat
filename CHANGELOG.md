- **2026-04-23** — Display bring-up round 15: fix the two wrong assumptions that kept the round-14 driver producing a black screen. Round 14 compiled clean, flashed fine, and every log line reported success — TCA9554 acked, 37-command ST7701 init sequence sent, esp_lcd_new_rgb_panel up at 14 MHz PCLK with a 480x480 PSRAM framebuffer, backlight bit set, LVGL alive — but the panel still showed nothing. A second read of the LovyanGFX discussion #630 config for this exact board (the same source we used for the RGB pin map) surfaced two off-by-one errors we'd been carrying since round 13. First: the backlight is not on the TCA9554 at all. It lives on RAW GPIO6 with PWM via Waveshare's ledcAttach(6, 20000, 10). Our own boot-log GPIO idle-state scan showed gpio 6 sitting LOW with nothing driving it, which is exactly what an unpowered backlight FET gate looks like — the clue was in our own logs, we just didn't know what to do with it. Second: the TCA9554 EXIO labels in the LGFX config are 1-indexed helper names, not direct bit numbers: EXIO1 is bit 0, EXIO2 is bit 1, EXIO3 is bit 2, EXIO8 is bit 7. Round 13 read them as direct bit numbers which shifted every signal up by one, so round 14's panel-reset pulse was actually hitting the touch-reset line and the ST7701 was never reset. It stayed in its power-on idle state and ignored the init sequence entirely, which explains both the dark screen and why the backlight bit we were toggling (TCA9554 IO0 = actually LCD_RST) made no visible difference. Round 15 fixes are all in display_pins.h and st7701_panel.cpp: the TCA9554 bit-mask constants shift down by one (bit 0 = LCD_RST, bit 1 = TP_RST, bit 2 = LCD_CS, bit 7 = BUZZER untouched because bit 7 is bit 7 either way which is the accidental reason round-12 beep detection produced the right buzzer mapping), TCA9554_BIT_LCD_BL is deleted entirely, new BACKLIGHT_PIN / BACKLIGHT_PWM_CH / BACKLIGHT_PWM_FREQ / BACKLIGHT_PWM_RES / BACKLIGHT_PWM_FULL constants describe the GPIO6 PWM setup, and St7701Panel::begin() replaces the TCA9554 backlight-enable call with ledcSetup(0, 20000, 10) + ledcAttachPin(6, 0) + ledcWrite(0, 1023). Arduino-ESP32 2.0.16 doesn't have the ledcAttach convenience API (that's 3.x) so we use the 3-function form. WIRING.md display and TCA9554 tables are regenerated with the corrected mapping, tca9554.cpp safe-state comment matches, and the display_pins.h header gains a round-15 pivot note that explains the mistake in full so future readers don't undo it. Next flash should paint the LVGL boot screen; if it's still dark, the remaining suspects in decreasing likelihood are (a) RGB timing porch values off, resolvable by widening h_back_porch / v_back_porch, (b) wrong ST7701 init sequence for this specific panel sub-variant, (c) RGB data-pin order wrong on one of R/G/B (would show as correct brightness but garbled colour, not black, so unlikely).

- **2026-04-23** — Display bring-up round 14: ship the real ST7701 RGB driver and retire the ST77916 QSPI stub. Round 13 established the chipset pivot on paper; round 14 turns that into running code. New files src/display/st7701_panel.h and st7701_panel.cpp implement the panel against ESP-IDF's esp_lcd_panel_rgb, which ships inside the Arduino-ESP32 2.0.16 IDF 4.4.7 fork so no toolchain bump is needed. Bring-up runs in six steps inside begin(): assert LCD_RST through TCA9554 IO1, release it and wait 120 ms for the ST7701 PLL, bit-bang the 37-command vendor init sequence over a 3-wire SPI link built from SCK=GPIO2 + SDA=GPIO1 + CS=TCA9554 IO3 using 9-bit transactions (one DCX bit then 8 data bits MSB-first), construct esp_lcd_new_rgb_panel with the full 16-bit RGB pin map and porch timings from display_pins.h at 14 MHz PCLK with a PSRAM-backed 480x480x2 framebuffer, call esp_lcd_panel_init to push the first frame and start PCLK, then raise LCD_BL on TCA9554 IO0 last so the first visible state is a blacked-out framebuffer rather than DRAM garbage. The init table is sourced from Nicolai-Electronics/esp32-component-st7701 vendor_specific_init_default[] — same BK0/BK1 gamma + GIP sequence that the Waveshare reference demo uses, just spelled out as a C array instead of hidden inside a vendor LCD_Driver component. drawBitmap() and fillColor() are thin wrappers over esp_lcd_panel_draw_bitmap so Ui.cpp's flush_cb keeps the identical signature it had for ST77916 — the only line that changed in Ui.cpp is the #include and class type swap from display::St77916Panel to display::St7701Panel plus a log-message update. st77916_panel.h and st77916_panel.cpp are deleted. If PSRAM framebuffer allocation fails (8 MB board but a runaway task could still exhaust it) begin() returns false and the LVGL flush becomes a no-op rather than crashing. First compile attempt also flagged a struct-field mismatch: cfg.bounce_buffer_size_px does not exist in IDF 4.4.7 — that field was added in IDF 5.x as part of the SRAM bounce-buffer fast path for high-PCLK PSRAM framebuffers. At 14 MHz we don't need it anyway, so the line is removed and replaced with a comment pointing at the IDF version pivot for future reference. Next flash should paint the LVGL boot screen; if it's still dark, suspects are: (a) IO0=LCD_BL hypothesis, verifiable by driving that bit high in isolation, (b) SPI init pins GPIO1/GPIO2 wrong on this board revision, verifiable by checking factory firmware pin usage, (c) timing porch values mismatched, resolvable by widening h_back_porch / v_back_porch.

- **2026-04-23** — Display bring-up round 12: TCA9554 IO-to-signal mapping diagnostic. Round 11 landed cleanly (TCA9554 ACK at 0x20, config+output writes accepted) but the backlight still never came on, which means our guess IO0=LCD_BL active-high is wrong for this board rev. Rather than guess again, brute-force it. New diagnostic block in src/Ui.cpp runs right after TCA9554.begin and before ST77916.begin stomps state: 10 phases, each held 2 s, logged with a distinctive phase letter. Phase A = all bits LOW, phase B = all bits HIGH, phases C..J = one bit HIGH walking IO0..IO7. Operator watches the screen and reports which phase letter(s) light the backlight. Decoding chart in the block comment: phase B plus one C..J means active-HIGH on that bit; phase A plus every C..J except one means active-LOW on the missing bit's letter; nothing at all means the backlight isn't wired through TCA9554 and round 13 pivots to a direct-GPIO probe. No bit mapping changed yet; display_pins.h's TCA9554_BIT_* constants stay as-is until we have the reported phase letter. Diagnostic adds ~20 s to boot; removed once we know the answer.

- **2026-04-23** — Display bring-up round 10: CHIPSET PIVOT. Round 9's exhaustive 420-pair I2C scan produced a clean result — only SDA=GPIO15 ever ACKs, and the only three addresses that respond are 0x15 (CST820 touch), 0x20 (TCA9554 I/O expander), and 0x51 (PCF85063 RTC). 0x24 and 0x38 — the CH422G MODE + WR_IO registers — never ACKed on any pin pair, meaning the board does NOT host a CH422G at all. That address signature (0x15 / 0x20 / 0x51) matches Waveshare's *other* 'ESP32-S3-Touch-LCD-2.1' BOM: ST7701 RGB panel (NOT QSPI ST77916) + TCA9554 expander (NOT CH422G) + CST820 + PCF85063. Every one of our last 9 rounds has been building drivers for the wrong chipset family. This round does the minimum useful thing with that information: (1) lock I2C pins in display_pins.h to SDA=GPIO15 / SCL=GPIO7 (GPIO7 is Waveshare's silkscreen convention on the 4' sibling board; it gets verified at boot), (2) document the chipset pivot inline with a pivot plan (round 11 = swap ch422g.cpp for TCA9554; round 12+ = replace ST77916 QSPI driver with esp_lcd_panel_rgb ST7701 + PSRAM framebuffer), (3) add TCA9554 register/bit definitions (TCA9554_REG_CONFIG=0x03, TCA9554_REG_OUTPUT=0x01, bits IO0=LCD_BL / IO1=TP_RST / IO2=LCD_RST / IO3=SD_CS) alongside the existing CH422G ones — the CH422G symbols stay temporarily so ch422g.cpp still links, but are clearly marked UNUSED, (4) swap round 9's 420-pair exhaustive scan for a targeted round-10 diagnostic that fixes SDA=GPIO15 and runs the full 127-address scan on each SCL candidate in {7, 1, 2, 5, 8, 9, 16}. The correct SCL will produce exactly three ACKs (0x15 / 0x20 / 0x51); wrong SCLs will drop ACKs or produce spurious ones. Once round 10's serial log confirms SCL=GPIO7 gives the clean 3-hit set, round 11 can rip out ch422g.cpp and move on to the ST7701 RGB rewrite. No driver code changes this round — purely diagnostic + pivot documentation.
